- Line 189: `vi.mocked(PdfParser.encode).mockReturnValueOnce(...).mockReturnValueOnce(...)`

**Source Usage** (demo/App.tsx, line 27):
```typescript
const result = await PdfParser.encode(file)
```

**Local Type Shim** (src/types/pdf-parser.d.ts):
```typescript
declare module '@hamster-note/pdf-parser' {
  import type { IntermediateDocument } from '@hamster-note/types'
  export const PdfParser: {
    encode(fileOrBuffer: File | ArrayBuffer): Promise<IntermediateDocument | undefined>
  }
}
```

**Analysis**: The PdfParser is declared as a plain object with an `encode` method (not a class). The mock correctly provides `PdfParser.encode` as a function. The 0.6.0 pdf-parser package exports `PdfParser` from the bundled build (pdfParser-Da5CndzI.js) and the types entry points to `dist/index.d.ts` which doesn't exist (build artifact issue), but the actual runtime behavior matches the mock.

Note: The local type shim says `Promise<IntermediateDocument | undefined>` but the document-parser base class (which PdfParser extends in 0.6.0) returns `Promise<IntermediateDocument>`. The test at line 114 that mocks `undefined` works because the mock bypasses the real implementation.

**Conclusion**: ✅ NO CHANGE NEEDED (for test mocks)

## Additional Observations

1. **Local Type Shim Drift** (NOT a test issue): The file `src/types/pdf-parser.d.ts` declares `PdfParser.encode` returning `Promise<IntermediateDocument | undefined>`, but the 0.6.0 document-parser base class returns `Promise<IntermediateDocument>`. This is a source-code type issue, not a test issue.

2. **Missing Type Declarations**: The pdf-parser 0.6.0 package.json declares `"types": "dist/index.d.ts"` but that file doesn't exist in the dist folder. This is a package build issue, not relevant to tests.

3. **Test Independence**: Tasks 5 (html-parser) and 6 (pdf-parser) can proceed independently since no changes are needed to either test suite.
